# /review-ui — AI駆動UIレビュー

UI変更に対するVisual Regression Testを自動実行する。
変更の種類を自動判別し、適切なフローで処理する。

---

## Step 1: 変更内容の把握と分類

1. `git diff origin/main --name-only` で変更ファイル一覧を取得
   - origin/mainがない場合は `git diff main --name-only` を試す
2. 変更ファイルの内容をReadで読み取り、何が変わったかを正確に把握する
3. 各変更ファイルを以下のカテゴリに分類する:

| カテゴリ | 判定条件 | 処理フロー |
|---------|---------|-----------|
| **A. 新規UI追加** | 新規の画面/コンポーネントファイルが追加されている | → ベースライン撮影フロー |
| **B. 共通コンポーネント/トークン変更** | Color.kt, Theme.kt, PrimaryButton.kt等の既存共通ファイルが変更されている | → 影響範囲分析フロー |
| **C. 画面固有の変更** | 特定の画面ファイルのみが変更されている | → 対象画面の差分検出フロー |
| **複合** | 上記が複数該当する | → 各カテゴリの処理を順に実行 |

---

## フローA: 新規UI追加

新しい画面やコンポーネントが追加された場合のフロー。

### A-1: 新規ファイルの確認

追加されたComposable関数/画面を特定する。

### A-2: スクリーンショットテストの有無を確認

`app/src/test/` 配下に、追加されたコンポーネント/画面に対応するスクリーンショットテストが存在するか確認する。

- **テストがある場合** → A-3へ
- **テストがない場合** → レポートに「スクリーンショットテストの追加を推奨」と記載

### A-3: ベースライン撮影

新規テストに対応するベースラインを撮影する。

```bash
./gradlew recordRoborazziDebug --tests "テストクラス名.新規テストメソッド"
```

### A-4: 撮影画像のレビュー

撮影されたベースライン画像をReadで読み取り、以下を確認:
- レイアウトが意図通りか
- テキストの可読性
- ダークテーマでの表示
- 既存コンポーネントとの一貫性

### A-5: レポート出力

```
## 🤖 AI駆動UIレビュー — 新規UI追加

### 追加されたUI
| コンポーネント/画面 | ファイル | テストカバレッジ |
|---|---|---|

### ベースライン画像
#### {コンポーネント名} (Light)
**レビュー**: （レイアウト・デザインの評価）

![コンポーネント名_light](画像パスまたはCDN_URL)

#### {コンポーネント名} (Dark)
**レビュー**: （ダークテーマでの評価）

![コンポーネント名_dark](画像パスまたはCDN_URL)

### 既存UIへの影響
（既存コンポーネント/画面への影響がないことを確認した旨を記載）

### 総合判定
（新規UIの品質評価とマージ可否の推奨）
```

## フローB: 共通コンポーネント/トークン変更

Color.kt、Theme.kt、共通コンポーネント（PrimaryButton, TopAppBar等）が変更された場合のフロー。
**影響範囲の特定がこのフローの核心。**

### B-1: 影響範囲の網羅的な特定

変更されたコードから依存関係を辿り、影響するコンポーネント・画面を**すべて**特定する。

#### 手順

1. 変更されたファイル内のpublic関数/プロパティ名を特定
2. それらの名前をGrepで検索し、直接使用しているファイルを見つける
3. さらにそのファイルを使用しているファイルをGrepで再帰的に辿る
4. 最終的に影響を受けるすべてのコンポーネントと画面をリストアップ

#### 判断基準の例

- Color.kt の `SurfaceLight` を変更した場合:
  - → Theme.kt の `LightColorScheme.surface` で使用
  - → `MaterialTheme.colorScheme.surface` を使う TopAppBar（containerColor）に影響
  - → TopAppBar を使う SampleScreen, SettingsScreen にも影響
  - → PrimaryButton は `primary` 色を使っているため影響**なし**
  - → PrimaryButton のみ使う LoginScreen も影響**なし**

**重要**: 「影響あり」だけでなく「影響なし」の根拠も必ず示すこと。

### B-2: 影響範囲に対応するテストを選定

B-1で特定した影響を受けるコンポーネント/画面に対応するスクリーンショットテストだけを選定する。

`app/src/test/` 配下のテストファイルを確認し、テストメソッド名とコンポーネント名の対応を確認する。

### B-3: 選定したテストだけ実行

```bash
./gradlew compareRoborazziDebug \
  --tests "テストクラス名.テストメソッド1" \
  --tests "テストクラス名.テストメソッド2"
```

**なぜ全テスト実行ではなく選択実行か**: AIが影響範囲を特定した結果がテスト選定という実際のアクションに直結する。これがAIを使う意味。

### B-4: 差分画像の収集とアップロード

```bash
find app/build/outputs/roborazzi -name "*_compare.png" 2>/dev/null | sort
```

CI環境（環境変数 `GITHUB_ACTIONS` または `REPO_TOKEN` が存在）では `cml publish` でCDNにアップロード:
```bash
cml publish "画像パス"
```

### B-5: 差分画像のデザインレビュー

差分画像をReadで読み取り、各画像について:

1. **変更前後の違い**: Reference（左）とNew（右）の具体的な差異
2. **意図との整合性**: コード変更の意図と見た目の変化が一致しているか
3. **デザイン品質**: レイアウト崩れ、コントラスト、ダークテーマでの視認性

#### 判定基準

- ✅ **意図した変更**: PRの目的に沿った見た目の変化。問題なし
- ⚠️ **要確認**: 意図した変更だが、コントラスト・アクセシビリティ等の懸念あり
- ❌ **レグレッション**: PRの目的と無関係な見た目の変化、またはレイアウト崩れ

**レグレッション判定の基準**: PRの目的（タイトル・説明・変更概要）と照らし合わせて、その差分がPRの意図した変更に起因するかどうかで判断する。例えば「サーフェスカラー変更」のPRでPrimaryButtonの高さが変わっていたらレグレッション。

### B-6: テストカバレッジ確認

影響を受けるコンポーネント/画面のうち、スクリーンショットテストがないものを特定する。

### B-7: レポート出力

```
## 🤖 AI駆動UIレビュー — 共通コンポーネント/トークン変更

### 変更概要
（何がどう変わったかを1-2行で）

### 影響範囲分析

#### ⭕ 影響あり
| コンポーネント/画面 | 影響の理由 | テストカバレッジ |
|---|---|---|

#### ➖ 影響なし（確認済み）
| コンポーネント/画面 | 影響なしの根拠 |
|---|---|

### 実行したテスト
以下のテストを影響範囲に基づいて選択的に実行しました:
- （テスト一覧と選定理由）

### スクリーンショット差分

#### {テスト名}
**判定**: ✅ / ⚠️ / ❌
**分析**: （変更前後の違い、デザイン観点の評価）

![テスト名](CDN_URLまたはローカルパス)

### 手動確認が必要な画面
テスト未カバーで影響を受ける画面のリスト。
すべてカバーされている場合は「すべての影響画面がテストでカバーされています」と記載。

### 総合判定
変更全体の安全性とマージ可否の推奨を1-3行で記載。
ベースライン更新が必要な場合は `./gradlew recordRoborazziDebug` の実行を案内。
```

## フローC: 画面固有の変更

特定の画面ファイルのみが変更された場合のフロー。影響範囲は限定的。

### C-1: 変更された画面の特定

変更されたファイルに対応する画面を特定する。

### C-2: 対応するテストを実行

変更された画面に対応するスクリーンショットテストだけを実行する。

```bash
./gradlew compareRoborazziDebug --tests "テストクラス名.対象テスト"
```

対応するテストがない場合は、レポートに「テスト追加を推奨」と記載。

### C-3: 差分画像のレビュー

フローBのB-5と同じ手順でレビュー。

### C-4: レポート出力

```
## 🤖 AI駆動UIレビュー — 画面修正

### 変更概要
（何がどう変わったかを1-2行で）

### 変更された画面
| 画面 | 変更内容 | テストカバレッジ |
|---|---|---|

### スクリーンショット差分

#### {テスト名}
**判定**: ✅ / ⚠️ / ❌
**分析**: （変更前後の違い、デザイン観点の評価）

![テスト名](CDN_URLまたはローカルパス)

### 他の画面への影響
（他の画面に影響がないことを確認した旨を記載。影響がある場合はフローBに切り替え）

### 総合判定
（安全性とマージ可否の推奨）
```

## 共通: 画像アップロード

CI環境（環境変数 `GITHUB_ACTIONS` または `REPO_TOKEN` が存在する場合）では、PRコメントに画像を直接表示するために `cml publish` でCDNにアップロードする。

```bash
CDN_URL=$(cml publish "画像パス")
```

レポート内では `![テスト名](CDN_URL)` の形式で埋め込む。

ローカル環境ではアップロードせず、ローカルパスをそのまま記載する。

## 共通: ベースライン確認

テスト実行前に必ずベースライン画像の存在を確認する:

```bash
ls app/screenshots/*.png 2>/dev/null | head -20
```

ベースラインが存在しない場合は `./gradlew recordRoborazziDebug` で撮影し、その旨をレポートに記載する。
