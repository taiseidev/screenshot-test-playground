# /review-ui — AI駆動UIレビュー

PRを出す前の「どの画面が影響する？見た目は大丈夫？」を全自動で実行する。

## Step 1: 影響範囲分析

git diff から変更内容を把握し、影響する画面を特定する。

1. `git diff develop --name-only` で変更ファイル一覧を取得
2. 変更ファイルの内容を Read で読み取り、何が変わったかを把握
3. 変更された関数/クラス/トークンを使用している箇所を Grep で検索
4. 依存チェーンを辿り、最終的に影響する画面（Screen/Activity）を特定
5. **影響しない画面とその理由**も明記する（例: colorResource経由のため影響なし）

### 出力フォーマット

```markdown
## 影響範囲分析

### 変更内容
- `ファイルパス`: 変更の概要

### 影響を受ける画面
| コンポーネント/画面 | ファイルパス | 影響の理由 |
|---|---|---|

### 影響を受けない画面（確認済み）
| コンポーネント/画面 | ファイルパス | 影響しない理由 |
|---|---|---|
```

## Step 2: スクリーンショット差分検出

影響範囲に基づいてRoborazziで差分を検出する。

1. ベースライン画像が `screenshots/` に存在するか確認
2. ベースラインがない場合: `./gradlew recordRoborazziDebug` でベースライン撮影
3. ベースラインがある場合: `./gradlew compareRoborazziDebug` で差分生成
4. `app/build/outputs/roborazzi/` 配下の `*_compare.png` を探す

## Step 3: 差分画像のAIレビュー

差分画像が見つかった場合、Read ツールで画像を読み取り分析する。

各差分画像について以下を判定:
- **変更前（Reference）** と **変更後（New）** の違い
- レイアウトの崩れがないか
- 色のコントラストが十分か（特にダークテーマ）
- コード変更の意図と一致しているか

### 判定基準
- ✅ **意図した変更**: コード変更の意図通りの見た目の変化
- ⚠️ **要確認**: 意図した変更だが確認推奨（コントラスト、アクセシビリティなど）
- ❌ **レグレッション**: 意図しない見た目の変化、レイアウト崩れ

## Step 4: レポート出力

以下のフォーマットでレポートを出力する。PRコメントにそのまま貼れる形式。

```markdown
# AI駆動UIレビュー結果

## 1. 影響範囲
（Step 1の結果）

## 2. スクリーンショット差分
| テスト名 | 差分有無 | 判定 |
|---|---|---|
| PrimaryButton_light | あり | ✅ 意図した変更 |
| PrimaryButton_dark | なし | - |

### 差分詳細
（差分があった画像ごとに分析結果を記載）

## 3. 総合判定
- ✅ / ⚠️ / ❌ の集計
- 手動確認が必要な画面のリスト（テスト未導入の画面）

## 4. 推奨アクション
- 確認が必要な項目のリスト
```
